# תוכנית: שכבת "ספר עבודה" אינטראקטיבית

מטרה: לכל יחידה באתר יתווסף אזור "ספר עבודה" שמממש דיגיטלית את המשימות של
Workbook 1 (יחידות 1–19 במהדורה השנייה) ו־Workbook 2 (יחידות 20–32),
כולל ממשק כתיבה בארבעה קולות עם בדיקה אוטומטית של הובלת קולות.

---

## 1. עיקרון תוכן (זכויות יוצרים)

ספרי העבודה מוגנים בזכויות יוצרים, בדיוק כמו הספר עצמו. לכן:

- **לא מעתיקים** את המלודיות, הבסים הממוספרים והתרגילים שהמחברים חיברו.
  כותבים תרגילים מקוריים **באותה רוח בדיוק**: אותו סוג משימה, אותה רמת קושי,
  אותם אילוצים ("השתמש רק ב־I, V, V7", "כלול לפחות VII6 אחד", "בתיבה 1 החלפת
  קולות בין סופרן לבס"), אותה כמות בקירוב.
- קטעי הספרות לניתוח (Study and Analysis) הם יצירות נחלת הכלל — מותר ואף רצוי
  להשתמש **באותן יצירות עצמן** (כוראלים של באך, סונטות של מוצרט ובטהובן וכו'),
  משוקעות מחדש מהתווים המקוריים, לפי הכלל הקיים: לא לנחש תיבות.
- הוראות המשימות נכתבות בעברית מקורית, לא כתרגום.

## 3. טקסונומיית המשימות → מימוש דיגיטלי

מקריאת ספרי העבודה עולים שישה טיפוסי משימה שחוזרים לכל אורכם:

### א. תרגילי יסודות (יחידות 1–4)
"כתוב סולם שלם מתוך צליל ודרגה", "מצא את הצליל מסולם+דרגה", "זהה סולם מסימני
היתק", מרווחים, משולשים.
**מימוש:** שני מנועים —
1. **StaffInput** — חמשה שבה התלמיד מניח צלילים (ראו §5): למשל נתון מי♭ כדרגה
   6̂, כתוב את כל הסולם המז'ורי על החמשה.
2. הרחבת ה־`Drill` הקיים לגרסה "סדרתית" (סט שאלות סגור עם התקדמות, לא רק
   אקראי אינסופי) — לשאלות זיהוי.

### ב. איתור ותיקון שגיאות (יחידה 6, וממשיך לאורך הספר)
"סמן שגיאות פריסה/הכפלה/טווח/מקבילות באקורדים הבאים; לא כולם שגויים. כתוב
גרסה מתוקנת."
**מימוש:** תצוגת SATB שבה כל אקורד/זוג אקורדים לחיץ; התלמיד מסמן היכן השגיאה
ובוחר את סוגה מרשימה (מקבילות אוקטבות, חריגת טווח, מרווח פנימי גדול מאוקטבה,
הכפלה שגויה…). שלב ב' אופציונלי: תיקון בפועל בעורך ה־SATB. הבדיקה כאן דטרמיניסטית
כי השגיאות ידועות מראש.

### ג. Preliminaries — קטעי חימום (מיחידה 7 והלאה)
פרגמנטים של 2–4 אקורדים: נתון סופרן (או בס ממוספר) + אילוץ ("רק I, V, V7").
**מימוש:** עורך ה־SATB (ראו §4) במצב קצר, עם בדיקת כללים חיה. זה לב הפרויקט —
רוב עמודי העבודה בנויים מזה.

### ד. משימות ארוכות — מלודיות ובסים
הרמוניזציה של מלודיה, מימוש בס ממוספר, בס לא ממוספר, שני קולות חיצוניים
נתונים, כוראל.
**מימוש:** אותו עורך SATB במצב מלא (6–10 אקורדים), עם הקולות הנתונים נעולים
והספרות מוצגות מתחת לבס. בנוסף שדה סימון דרגות: התלמיד מסמן לכל עמדה גם ספרה
רומית (בחירה מרשימה), כך שהבדיקה יודעת לְמה הוא התכוון ומדווחת גם "האקורד לא
תואם את הספרה שבחרת".

### ה. ניתוח יצירות (Study and Analysis)
קטע מהספרות, משוקע ומתנגן; התלמיד מסמן ספרה רומית + ספרות לכל אקורד, ועונה
על שאלות מודרכות ("כמה זמן נמשכת הטוניקה של תיבה 4?").
**מימוש:** קומפוננטת **AnalysisExercise** — Score קיים + שורת תיבות בחירה
מיושרת מתחת לאקורדים (בחירה מתפריט מצומצם פר־אקורד, כמו ב־Quiz הקיים) + שאלות
אמריקאיות. השמעה עם הדגשה כרגיל.

### ו. משימות "נייר חופשי" שאין טעם לדגטל
הוראות כמו "כתוב וריאציה", "המשך את הקטע בסגנון" — יישארו כהצעות עבודה
פתוחות בסוף עמוד העבודה ("למחברת"), בלי בדיקה. לא מנסים לכפות בדיקה על מה
שדורש אוזן של מורה.

## 4. הרכיב המרכזי: SatbEditor

### עקרון העיצוב
עורך על גבי חמשה כפולה אמיתית (VexFlow, אותה שיטת שיקוע כמו `Satb.tsx`
הקיים), לא ממשק מופשט. עריכה בשכבת SVG מעל התווים. RTL בטקסט, אבל התווים
עצמם LTR כרגיל.

### מודל הנתונים
```ts
type Voice = "s" | "a" | "t" | "b";
type Slot = { chordIndex: number; voice: Voice };

type SatbExercise = {
  id: string;                       // "u07-prelim-3"
  kind: "satb";
  key: string;                      // "Eb" / "c" (מינור באות קטנה)
  meter?: [number, number];
  length: number;                   // מספר אקורדים
  given: Partial<Record<Voice, (Pitch | null)[]>>;  // קולות נתונים ונעולים
  figures?: (string | null)[];      // בס ממוספר להצגה: "6", "65", "#", …
  romanSlots?: boolean;             // האם התלמיד מסמן דרגות
  constraints: Constraint[];        // "רק I,V,V7", "כלול VII6", "החלפת קולות בתיבה 1"
  instructions: string;             // ההוראה בעברית
  modelSolution?: SatbChord[];      // פתרון לדוגמה (מקורי שלנו) להצגה אחרי בדיקה
};
```
`Pitch` = `{ step, alter, octave }` (או midi + כתיב); השוואות אנהרמוניות לפי
chroma, כתמיד.

### אינטראקציה (מהירה, מקלדת־קודם)
- **סלוט פעיל**: תא = קול × עמדה. ניווט בחיצים (→/← עמדה, ↑/↓ קול) ובקליק.
  הסלוט הפעיל מסומן בצבע; קולות נתונים נעולים ומעומעמים קלות.
- **הזנת גובה**, שלוש דרכים מקבילות:
  1. הקלדת אות (A–G) — הצליל נכנס באוקטבה הקרובה ביותר לטווח הקול; Shift+↑/↓
     מזיז אוקטבה; `+`/`-`/`0` דיאז/במול/בקאר.
  2. קליק על גובה בחמשה (אזורי snap לקווים ולרווחים) — כמו להניח תו בעיפרון.
  3. הפסנתר הקיים (`Keyboard`) מתחת לעורך — קליק על קליש מזין את הסלוט הפעיל
     ומתקדם אוטומטית לסלוט הבא.
- Backspace/Delete מוחק; Tab עובר לאקורד הבא.
- כל הזנה מנוגנת מיד (playNote) — פידבק שמיעתי בלי לחכות.
- **מובייל**: העורך יעבוד גם במגע (קליק על חמשה/פסנתר), אבל הוא מוצהר
  כחוויה שולחנית בעיקרה; לא מתפשרים על העיצוב בגללו.

### שיקוע
- שימוש חוזר בלוגיקת `SatbScores` (חמשה כפולה, brace, יישור פורמטר אחד,
  התאמת ערכת נושא), בתוספת:
  - שכבת hit-areas שקופה מעל כל עמדה לצורך קליקים,
  - צביעת הסלוט הפעיל, צביעת שגיאות (ראו למטה),
  - סימני היתק לפי הסולם (`accidentalKey` כבר קיים),
  - ספרות בס מתחת לחמשה התחתונה, ספרות רומיות של התלמיד מתחתן.
- רה־רינדור מלא של ה־SVG בכל שינוי state — בגדלים האלה (עד ~10 אקורדים)
  זה זול, וזה הדפוס הקיים בקוד.

### מנוע הבדיקה: `src/engine/voiceLeading.ts`
פונקציה טהורה, ניתנת לבדיקה ביחידות:
```ts
type Violation = {
  rule: RuleId;
  severity: "error" | "warning";
  slots: Slot[];                 // מה לצבוע
  message: string;               // עברית, מנוסח פדגוגית
};
checkSatb(chords, exercise): Violation[]
```
כללים (מצטברים לפי היחידה — ראו "פרופיל כללים"):
1. טווחי קולות (S: דו4–סול5, A: סול3–רה5, T: דו3–סול4, B: מי2–דו4, בקירוב הספר).
2. מרווח בין קולות עליונים ≤ אוקטבה; בס חופשי.
3. הצלבת קולות ו־overlap בין אקורדים סמוכים.
4. שלמות האקורד והכפלות (לפי ההרמוניה המזוהה/המוצהרת).
5. קווינטות ואוקטבות מקבילות, וכן אנטי־מקבילות.
6. קווינטות/אוקטבות סמויות אל הקולות החיצוניים (warning).
7. פתרון צליל מוביל (בקול חיצוני — חובה ל־1̂; פנימי — מותרת הקפיצה, warning).
8. פתרון ספטימה כלפי מטה; הכנת ספטימה היכן שנדרש (II7 וכו').
9. 2̂מוגדלת/קפיצות אסורות במינור (הימנעות מסקונדה מוגדלת).
10. התאמה להרמוניה: אם יש figures — צלילי האקורד חייבים לממש את הספרות; אם
    יש romanSlots — התאמה בין מה שנכתב למה שסומן.
11. אילוצי תרגיל (constraints): אוצר אקורדים מותר, אירוע נדרש ("לפחות VII6
    אחד"), קדנצה בסוף.

**פרופיל כללים פר־יחידה**: יחידה 7 בודקת 1–7 בלבד; ספטימות מצטרפות ביחידה 9;
וכן הלאה. כך תלמיד לא ננזף על חומר שטרם נלמד. הפרופיל מוגדר בנתוני התרגיל.

### חוויית הבדיקה
- כפתור **"בדיקה"** (ברירת מחדל) — לא בדיקה חיה תוך כדי הקלדה, כדי לא להעניש
  מצבי ביניים. אחרי בדיקה: הסלוטים הבעייתיים נצבעים (אדום=error,
  ענבר=warning), ומתחת לעורך רשימת ממצאים בעברית, כל ממצא לחיץ ומהבהב את
  הסלוטים שלו.
- מתג "רמזים חיים" למי שרוצה בדיקה תוך כדי כתיבה.
- כשהכול נקי: חיווי הצלחה + השמעת הפתרון + כפתור "פתרון לדוגמה" (מוצג לצד
  הפתרון של התלמיד, לא במקומו) — ברוח הערת המחברים שכדאי לכתוב פתרונות רבים
  ולהשוות.
- לתרגילי הרמוניזציה אין "תשובה נכונה" יחידה — הבדיקה היא אילוצים, לא השוואה
  לפתרון.

## 5. רכיב משני: StaffInput (חמשה בודדת)

גרסה מפושטת של אותו עורך: חמשה אחת, קול אחד, n עמדות. משרת את תרגילי
היסודות (כתיבת סולמות, מרווחים, בניית משולשים וספטאקורדים מעל בס נתון).
נבנה כאותו רכיב פנימי עם קונפיגורציה שונה — לא שני מימושים.

## 6. ארכיטקטורה באתר

```
src/workbook/
  types.ts             // Exercise: איחוד ממויין לפי kind
  Workbook.tsx         // עמוד עבודה של יחידה: רשימת תרגילים + התקדמות
  SatbEditor.tsx       // העורך (וגם מצב StaffInput)
  AnalysisExercise.tsx // ניתוח קטעי ספרות
  ErrorHunt.tsx        // איתור שגיאות
  DrillSet.tsx         // סט שאלות סגור
  progress.ts          // שמירה ב-localStorage
src/engine/
  voiceLeading.ts      // מנוע הכללים + בדיקות Vitest מקיפות
  pitch.ts             // ייצוג גובה מדויק (step/alter/octave) אם חסר
src/units/unitNN/
  workbook.ts          // נתוני התרגילים של היחידה (מקוריים!)
```

- **ראוטינג**: `#/unit/NN/workbook`. בעמוד היחידה, אחרי ה־Quiz הקיים, קלף
  "ספר עבודה" עם תקציר התקדמות; וכניסה גם ממפת הלמידה (אייקון עט ליד יחידה
  שיש לה workbook). הכלל "הוספת יחידה לא נוגעת ביחידות אחרות" נשמר: הרישום
  ב־registry יקבל שדה אופציונלי `workbook?: () => Promise<Exercise[]>`.
- **התקדמות**: localStorage — לכל תרגיל: מצב (לא התחיל/בעבודה/הושלם), והפתרון
  האחרון של התלמיד (כדי לחזור אליו). מפתח: `hl-wb-<unitId>-<exId>`.
- **עיצוב**: אותה שפה קיימת (widget cards, גופן, צבעי ‎--stable/--accent/--gold).
  תרגיל אחד פתוח בכל רגע (אקורדיון), כדי שהעמוד לא יהיה קיר של חמשות.

## 7. שלבי ביצוע

כל שלב נגמר במצב עובד, נבדק ומועלה (הכלל הקיים: commit+push אחרי כל שינוי עובד).

1. **שלב 0 — תשתית** (קטן): types, ראוט, עמוד Workbook ריק עם תרגיל דמה,
   progress.ts, כניסות מה־UI. 
2. **שלב 1 — מנוע הובלת קולות**: `voiceLeading.ts` + חבילת בדיקות Vitest
   (המנוע נכתב לפני העורך — הוא הליבה, וקל לבדוק אותו בלי UI).
3. **שלב 2 — SatbEditor MVP**: שיקוע + סלוטים + הזנה בשלוש הדרכים + בדיקה +
   השמעה. עמוד עבודה מלא ליחידה 07 (המקבילה ל־WB §6) כפיילוט: Preliminaries
   מקוריים + 2 משימות ארוכות + הרמוניזציה במינור מקביל.
4. **שלב 3 — StaffInput ותרגילי יסודות**: עמודי עבודה ליחידות 01–04 + 06
   (כולל ErrorHunt של יחידה 6).

## 8. סיכונים והחלטות פתוחות

- **ריתמוס בעורך**: MVP עובד באקורדים הומו־ריתמיים (כמו רוב תרגילי הספר —
  half/whole notes). מלודיות עם ריתמוס מגוון: בשלב ראשון הריתמוס נתון מראש
  בתרגיל (התלמיד ממלא גבהים בלבד), לא עורך ריתמוס חופשי. זה מכסה ~90% מהספר
  ושומר על העורך פשוט.
- **קדמות (anacrusis) ומשקלים**: נתמכים כנתוני תצוגה, לא כאובייקט עריכה.
- **זיהוי אקורד אוטומטי** בהרמוניזציה חופשית (בלי romanSlots): נדרש ניחוש
  ההרמוניה מהצלילים — נכון יותר לחייב את התלמיד לסמן דרגות, גם פדגוגית.
  החלטה: בתרגילי הרמוניזציה — סימון דרגות חובה.
- **אימות המיפוי** בטבלה שבסעיף 2 ייעשה מחדש מול ה־PDF בכל יחידה כשמגיעים
  אליה (בפרט 12–14 ו־20–23, שם המהדורות מתפצלות).
